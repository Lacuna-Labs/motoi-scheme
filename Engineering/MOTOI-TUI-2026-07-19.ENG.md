# Motoi Scheme TUI — 4-region terminal IDE mirroring the web IDE

**Version:** 1.0
**Date:** 2026-07-19
**Status:** landed
**Audience:** Marcus (infra); Alfred; downstream Sakura team when Curator TUI mirror lands

> **Provenance.** Alfred asked 2026-07-19: "Can I have the same thing on the TUI also? And!! Use Sakura's color palette. So it's all soft colors. Use it for our colors." Marcus shipped the 3-panel web IDE at `motoi ide` plus the Wave 2 pair-programming / reading-state / book tutor stack; Alfred asked to mirror the whole thing into a terminal-hosted 4-region layout with the Sakura palette applied everywhere.

## The shape

`motoi tui` launches an in-terminal 4-region IDE:

- Menu bar + cherry-tree stripe row at the top
- Project tree on the left (books · chapters · new-buffer)
- Tabbed editor in the center-right (buffers · chapter views · settings · help)
- REPL panel at the bottom
- Optional CPU panel between editor and REPL (F4 toggles)
- Status ribbon at the very bottom (mint band with cedar ink)

Feature parity with the web IDE is the north star. Same runtime — one `Session` per launch, built on `makeCoreEnv`, exposing every `book/*`, `cpu/*`, `motoi/pair-*`, `motoi/reading-state` verb the browser IDE reaches. Different view; same brain.

## The architecture

Five new files in `tui/`:

| File               | Purpose                                                                                  |
|--------------------|------------------------------------------------------------------------------------------|
| `tui/palette.js`   | Cherry-tree RGB tiers + 256-color fallback slots + `detectColor()` mirror of `src/repl.js`. |
| `tui/screen.js`    | Double-buffered cell grid · diff-flush · Unicode box drawing · alt-screen enter/leave. Zero deps. |
| `tui/input.js`     | Raw-mode keystroke reader with ANSI decode (arrows · F-keys · Alt+char · Ctrl+char · CSI-modified keys). |
| `tui/session.js`   | Thin wrapper over `makeCoreEnv` sharing the exact env shape with `src/ide-server.js`. Exposes `evalSource`, `bookList`, `bookToc`, `bookChapter`, `cpuDisplay`, `pairState`, `explain`, `ambientComplete`, `highlight`. |
| `tui/tui.js`       | The orchestrator. Module-scope state singleton with `_resetForTests()` teardown seam. Handles input dispatch (basic + vim + emacs modes) · layout · painting · menu-bar dropdowns · tabs · ghost completion · status flash. |

Wired into `src/cli.js` — `motoi tui` is a first-class subcommand. No new external dependencies. `package.json` unchanged.

## Vim + Emacs

**Vim** starts in NORMAL. Normal-mode keys:

```
h j k l   move
0 $       line ends
i a o     enter INSERT (append / o = open line)
x         delete char
:         open COMMAND line — :w :q :wq :explain :pair-on :pair-off
                              :cpu :help :settings
```

**Emacs** bindings compose with basic:

```
C-x C-s   save marker (in-session)
C-x C-c   exit
C-x C-e   eval enclosing form
M-e       explain enclosing form
M-x       eval enclosing form (short form)
```

**Basic** is what a kid uncomfortable with vim/emacs gets: arrow keys, printable text, `Ctrl+Enter` runs the enclosing form, `F2` explains selection.

## Key decisions

**1. Custom ANSI vs blessed/ink.** blessed is unmaintained; neo-blessed pulls a C build; ink pulls React + JSX. Motoi's `package.json` has zero runtime dependencies — adding one for the TUI would tell a bad story. The custom stack is ~15 KB across five files · zero deps · fully testable in-process with mocked stdout/stdin.

**2. Module-scope state singleton.** The web IDE keeps state in the browser's `window` object; a TUI process serves one user at a time so the module-scope singleton is fine. `_resetForTests()` gives the harness a clean seam.

**3. Diff-flush over full-clear.** A cleared screen flashes visibly on macOS Terminal + iTerm2 at 60 fps. Diff-flush emits ANSI + text only for cells that actually changed since the previous frame — average delta on a keystroke is ~40 cells.

**4. Cherry-tree metaphor for tiers.** Pink · mint · cedar · cream · pearl reads as one thing (a tree) rather than four independent colors. Every panel got a role assigned by the metaphor: parens = trunk (cedar); comments = leaves (mint-dark); strings = blossoms (pink-dark); status ribbon = leaf band (mint bg); active-tab underline = blossom bar (pink).

**5. F-keys > menu-driven.** The web IDE relies on a button row; the terminal has F1..F12 free plus `Alt+mnemonic` plus Tab focus cycling. `F1` help / `F2` explain / `F3` settings / `F4` CPU / `F5` pair / `F6` focus is a shape every kid who used a Speak-and-Spell or a DOS TUI recognises.

**6. Same-runtime, not RPC.** The TUI process embeds a `Session` (`makeCoreEnv`) — it doesn't talk HTTP to a separate motoi-scheme process. `(define x 5)` at the REPL is visible to the editor buffer eval AND the CPU (which reads `env.get('cpu/state')`). Single process, single env, no marshalling.

**7. Test-mode opts.** `startTui(opts.noBlink)` + `opts.noSignals` is a small ugliness but the alternative — leaky `setInterval` + `process.on('SIGINT')` across 17 tests — hung the test runner. Explicit test-mode opts keep prod clean AND tests hermetic.

## For Alfred

Fire up a terminal, `motoi tui`, and try:

- Tab a few times to cycle focus tree → editor → repl
- **F1** for the full key map
- **F5** to toggle pair mode; then type at the editor and wait ~3s for a ghost
- **F4** to open the CPU panel; type `(cpu/boot!)` at the REPL and watch it wake
- **Alt-S** to open Settings — pick vim mode, hit `i`, type Scheme, hit `Esc`, then `:w`
- **F2** anywhere on a form to make Motoi explain it in the REPL
- **^C** to leave (or `,exit` at the REPL)

Both browser + terminal now share the Sakura Cherry Tree colors. Same brain, softer light in both windows.
